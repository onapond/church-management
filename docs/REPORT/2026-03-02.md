## 2. 주요 작업 내용

### 2.1. 보고서 중복 제출 버그 수정 (`ReportForm.tsx`)
*   **문제**: 기존에는 '부서 + 날짜' 기준으로만 중복을 체크하여, 한 부서 내에서 한 셀장이 보고서를 올리면 다른 셀장들이 동일한 날짜에 보고서를 제출할 수 없는 버그 발생.
*   **해결**:
    *   `cell_leader` (셀장 보고서)의 경우: **'부서 + 날짜 + 셀 ID'** (또는 셀 미선택 시 작성자 ID)를 복합적으로 확인하여 중복을 체크하도록 로직 변경.
    *   기타 전문 보고서 (모임, 교육, 심방, 프로젝트 등): **'부서 + 날짜 + 작성자 + 제목'**을 기준으로 체크하여 여러 건 제출 허용.
*   **효과**: 동일 부서 내 여러 셀장들이 충돌 없이 각자의 보고서를 제출할 수 있게 됨.

### 2.2. 셀장 보고서 결재 프로세스 간소화 (`ReportDetail.tsx`, `approvals.ts`, `dashboard.ts`)
*   **문제**: 기존에는 모든 보고서가 4단계(제출 -> 협조 -> 결재 -> 확인) 결재 라인을 타게 되어 있어, 수많은 셀장 보고서가 상위 결재자(회장, 부장)의 대기 목록에 쌓여 업무 부하를 가중시킴.
*   **해결**:
    *   `cell_leader` 보고서는 **팀장(부서장)의 승인만으로 즉시 '최종 승인(final_approved)'** 처리되도록 권한 로직(`checkApprovalPermission`) 수정.
    *   팀장이 본인 소속 부서의 셀장 보고서만 결재할 수 있도록 부서 일치 여부(`reportDeptId`) 추가 검증.
    *   상위 결재자(회장, 부장 등)의 결재 대기 쿼리(`usePendingReports`, `useDashboardPending`)에서 `neq('report_type', 'cell_leader')` 조건을 추가하여 목록에서 제외.
    *   `cell_leader` 보고서 상세 페이지 UI에서 복잡한 4단계 결재선 대신 '셀장 제출 -> 팀장 확인' 2단계 심플 UI 적용.

### 2.3. 셀장 열람 권한 정교화 (`permissions.ts`, `ReportListClient.tsx`)
*   **문제**: 부서 내 모든 보고서(기타 모임, 교육 등)가 셀장에게 노출됨. 보안 및 정보의 과다 노출 문제 제기.
*   **해결**:
    *   `canViewReport` 함수 수정: 셀장(`role=team_leader`, `is_team_leader=false`)은 **소속 부서 내의 '주차 보고서(weekly)'와 '셀장 보고서(cell_leader)'만 열람 가능**하도록 제한. (모임, 교육, 심방 등은 본인 작성 건 외에 열람 불가)
    *   목록 페이지(`ReportListClient.tsx`)에서 필터링 시 `report_type`을 함께 전달하여 목록에서도 정확히 가려지도록 반영.

### 2.4. 사용자 편의성(UX) 개선
*   **대시보드**: 대시보드 '보고서 작성' 버튼 클릭 시, 부서장(`is_team_leader=true`)은 '주차 보고서'가 기본 선택되고, 일반 셀장(`is_team_leader=false`)은 '셀장 보고서'가 기본으로 선택되어 화면이 열리도록 개선 (`src/app/(dashboard)/reports/new/page.tsx`, `DashboardContent.tsx`).

## 3. 결과 및 향후 계획
*   **결과**: 셀장들의 보고서 제출이 원활해졌으며, 결재 라인 분리를 통해 상위 관리자의 피로도를 낮추고 데이터 접근 권한을 명확히 함.
*   **향후 계획**: 보고서 데이터가 안정적으로 쌓이는지 모니터링하고, 추후 통계 페이지에서 보고서 유형별(주차 vs 셀장) 집계가 정확히 이루어지는지 추가 확인 필요.
