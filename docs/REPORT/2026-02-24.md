# 배포 보고서 - 2026-02-24

## 1. 개요
오늘 작업은 보고서 관리 시스템의 안정성 강화, 교인 및 사용자 관리 기능의 버그 수정, 그리고 심방 기능의 고도화 및 보안 정책(RLS) 적용을 중심으로 진행되었습니다.

## 2. 주요 수정 및 구현 사항

### A. 보고서 관리 시스템 (Reports)
- **권한 세분화**: 작성자(팀장)가 본인의 '임시저장' 또는 '반려' 상태 보고서를 직접 삭제할 수 있도록 개선했습니다. (`canDeleteReport`, `canEditReport` 신규 도입)
- **중복 생성 방지**: 주차 보고서뿐만 아니라 모임, 교육, 셀장, 프로젝트 보고서도 동일 날짜/부서에 중복 생성되지 않도록 체크 로직을 강화했습니다.
- **결재 정보 초기화**: 반려된 보고서를 수정 후 재제출할 때, 이전 결재 기록 및 시간을 초기화하여 새로운 결재 흐름이 시작되도록 수정했습니다.
- **리소스 최적화**: 
  - 보고서 삭제 시 Supabase Storage(`report-photos`)의 관련 이미지 폴더를 자동 삭제하여 스토리지 누수를 방지했습니다.
  - 인쇄 시 생성되는 임시 `iframe`의 메모리 누수 문제를 해결했습니다.

### B. 교인 관리 (Members)
- **스토리지 파일 정리**: 교인 삭제 시 해당 교인의 프로필 사진 파일을 스토리지에서 즉시 삭제하도록 수정했습니다.
- **수정 로직 안정화**: 교인 정보 수정 시 사진 파일 명명 규칙을 개선하고, 부서 정보 업데이트 시 데이터 유실 위험을 줄이기 위해 로직 순서를 최적화했습니다.

### C. 심방 기능 (Visitations)
- **관리자 권한 확대**: 관리자(super_admin, president 등)도 모든 심방 일정을 수정 및 삭제할 수 있도록 기능을 열어주었습니다.
- **보안 정책(RLS) 적용**: `visitations` 테이블에 RLS 정책을 적용하여 보안을 강화했습니다.
- **사용성 개선**: 심방 등록 폼에서 부서 변경 시 선택된 교인이 불필요하게 초기화되는 현상을 개선했습니다.

### D. 사용자 관리 (User Management)
- **데이터 무결성 수정**: 관리자 페이지에서 사용자의 부서를 변경할 때 기존 '팀장 권한'이 초기화되던 버그를 수정했습니다. 또한 '부서 없음' 선택 시 DB에 정상 반영되도록 개선했습니다.

## 3. 후속 권고 사항
- **DB RLS 정책 직접 실행**: 코드상에 정책을 반영했으나, 실제 Supabase Dashboard의 SQL Editor에서 `supabase/rls-policies.sql`에 기술된 최신 정책을 한 번 더 실행해 주시기를 권장합니다.
- **회계 잔액 재계산**: 현재 회계 잔액은 클라이언트에서 계산되어 저장됩니다. 향후 데이터가 많아질 경우 DB 트리거를 통한 서버측 재계산 아키텍처로의 전환을 고려해볼 수 있습니다.
